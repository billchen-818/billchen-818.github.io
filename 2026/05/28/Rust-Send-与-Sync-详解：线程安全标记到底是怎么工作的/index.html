Rust Send 与 Sync 详解：线程安全标记到底是怎么工作的

2026-05-28
  • #Rust

很多人第一次接触 Rust 并发时，都会很快碰到两个名字：

  • Send
  • Sync

然后通常会顺手记住一句话：

  • Send 表示可以在线程间发送
  • Sync 表示可以在线程间共享

这当然不算错。

但如果只停在这两句口号上，后面很快还是会混：

  • 为什么 Rc<T> 不行，Arc<T> 却可以
  • 为什么 Mutex<T> 看起来明明是“可变的”，却又能是 Sync
  • 为什么很多类型既是 Send 也是 Sync，但这两个 trait 又不能混成一个
  • 为什么它们明明没方法，却能在并发里起这么大作用

这说明：

Send 和 Sync 真正难的地方，不在定义本身，而在于你是否分清了“跨线程转移值”和“跨线程共享引用”是两件不同的事。

这一篇我会围绕 4 个问题展开：

  1. Send 和 Sync 到底分别在约束什么
  2. 为什么 Rust 要把它们拆成两个标记 trait
  3. Rc、Arc、Mutex、RefCell 这些常见类型到底各自落在哪一边
  4. 这些规则为什么能在编译期挡住数据竞争

文中的完整示例工程在 workspaces/send-sync-demo。

一、先说结论：Send 管“值能不能过线程边界”，Sync 管“引用能不能过线程边界”

这是整篇文章最重要的一句话。

很多人一开始会把 Send 和 Sync 都理解成“线程安全”。

这虽然方向没错，但太粗。

更准确的理解应该是：

  • Send：一个值的所有权，能不能安全地从一个线程转移到另一个线程
  • Sync：一个类型的共享引用 &T，能不能安全地在多个线程之间共享

所以它们其实在回答两个完全不同的问题：

  1. 这个值能不能被搬走
  2. 这个值的引用能不能被多人同时看见

一旦把这两个问题分开，很多原本容易混淆的地方都会自然很多。

二、为什么 Rust 要把它们拆成两个 trait？

因为“可以移动”不等于“可以共享引用”，反过来也一样。

我们先看最直观的两种并发场景。

2.1 场景一：把值整个交给另一个线程

比如通过 thread::spawn(move || ...)，把某个值 move 到子线程里。

这时候原线程不再继续使用它，问题核心是：

这个值被另一个线程接管后，会不会出事？

这就是 Send 在管的事情。

2.2 场景二：多个线程一起持有同一个值的引用

比如几个线程共同访问一份共享状态。

这时问题核心变成：

即使这些线程没有拿走所有权，只是同时看这份值，会不会出事？

这就是 Sync 在管的事情。

所以 Rust 把它们拆开，不是为了增加概念，而是为了把两类完全不同的风险分开建模。

我自己的理解是：

Send 关注的是“值如何换线程”，Sync 关注的是“引用如何跨线程同时存在”。

这个区分非常本质。

三、它们为什么没有方法，却还能这么重要？

因为 Send 和 Sync 都是标记 trait（marker trait）。

也就是说，它们的作用不是提供行为方法，而是向编译器声明一种性质：

  • 这个类型满足某种线程安全前提

所以它们的价值不在“你能调用什么方法”，而在：

编译器会基于这些性质，决定某段并发代码是否允许通过类型检查。

例如你把一个值 move 到 thread::spawn 的闭包里时，编译器会检查它是不是 Send。

如果不是，代码在编译期就会被拒绝。

这正是 Rust 并发模型最强的一点：

很多线程安全问题，不是靠运行时兜底，而是靠类型系统提前挡掉。

四、先看 Send：什么叫“值能安全地跨线程移动”？

示例工程 workspaces/send-sync-demo/src/main.rs 里先演示了一种最简单的 Send 场景：

let (tx, rx) = mpsc::channel();
let sender = thread::spawn(move || {
    tx.send("hello from worker".to_string()).unwrap();
});
sender.join().unwrap();
println!("channel recv => {}", rx.recv().unwrap());

这里 tx 被 move 进了子线程。

这件事之所以成立，前提就是：

这个值转移到另一个线程以后，仍然能保持正确语义。

这就是 Send 最朴素的含义。

4.1 Send 不是说“这个值适合多线程共享”

这是很容易误解的一点。

Send 只是在说：

  • 这个值可以被另一个线程接手

它并不自动意味着：

  • 多个线程可以同时共享它的引用

也就是说：

Send 管的是所有权转移，不是共享访问。

五、再看 Sync：什么叫“共享引用能安全地跨线程存在”？

一个很经典的定义是：

如果 &T 是 Send，那么 T 就是 Sync。

这句话刚看很绕，但换成人话就是：

如果一个类型的共享引用可以安全地被送到别的线程，那这个类型就可以被认为是 Sync。

这里要抓住的是“共享引用”三个字。

Sync 从来不是在说：

  • 这个值自己一定不可变

而是在说：

  • 即使多个线程同时拿到它的共享引用，也不会因为没有同步机制而出事

这就是为什么 Mutex<T> 虽然内部允许可变访问，却仍然可以是 Sync。

因为它已经把并发访问的同步规则包进去了。

六、为什么 Rc<T> 不行，而 Arc<T> 可以？

这是最适合用来讲清 Send/Sync 的一组对照。

示例工程里保留了一段 Rc 对照：

let local_only = Rc::new(String::from("rc is not send"));
println!("rc strong count on main thread => {}", Rc::strong_count(&local_only));

// Rc<T> 既不是 Send，也不是 Sync。
// 如果把它 move 到 thread::spawn 里，代码会直接编译失败。

为什么 Rc<T> 不行？

因为 Rc 的引用计数更新不是原子的。

这意味着：

  • 如果多个线程同时操作同一个 Rc 的引用计数
  • 就可能发生竞争条件

所以它既不适合：

  • 把值安全地跨线程传递
  • 也不适合把共享引用安全地跨线程共享

而 Arc<T> 则不同。

Arc 的引用计数是原子操作，所以它为跨线程共享提供了最基础的计数安全保障。

但这里要非常注意一点：

Arc<T> 只是把“引用计数”做成了线程安全，不代表 T 内部就自动线程安全。

这句话非常重要。

很多人会把 Arc<T> 误解成“线程安全万能壳”，其实不是。

如果 T 本身不满足对应约束，Arc<T> 也救不了它。

七、为什么 Arc<Mutex<T>> 常常是对的？

示例工程里有一段共享状态代码：

let shared = Arc::new(Mutex::new(vec![1, 2, 3]));
let mut workers = Vec::new();

for value in [4, 5] {
    let shared = Arc::clone(&shared);
    workers.push(thread::spawn(move || {
        let mut guard = shared.lock().unwrap();
        guard.push(value);
    }));
}

最后输出：

shared vec => [1, 2, 3, 4, 5]

这一组组合为什么常见？

因为它刚好把两类问题分开解决了：

  • Arc 负责跨线程共享拥有权
  • Mutex 负责多线程下的互斥访问

也就是说：

Arc 解决“谁都能持有它”，Mutex 解决“但同一时刻只能有一个人改它”。

这就是它为什么会成为共享可变状态的经典组合。

7.1 Mutex<T> 为什么能是 Sync

很多人第一次接触这里会觉得有点反直觉：

既然 Mutex<T> 里面可以改数据，为什么它还能被共享？

因为 Sync 从来不是说“永远不能改”，而是说：

多线程共享时，类型自身已经提供了正确的同步机制。

Mutex 恰恰就是把“修改前必须拿锁”这件事变成了类型规则。

所以它可以安全地被共享引用跨线程持有。

八、那 RefCell<T>、Cell<T> 为什么常常和 !Sync 放在一起？

因为这类 interior mutability 类型的可变性检查，主要是为单线程场景设计的。

例如 RefCell<T> 的借用规则是在运行时检查的，但它本身并没有内建多线程同步机制。

这意味着：

  • 在单线程里，它很有用
  • 但如果跨线程共享，就可能出事

所以这类类型通常不适合作为 Sync 的基础。

这也再次说明：

“能修改内部值”和“能安全跨线程共享”完全不是同一回事。

如果缺少同步原语，仅靠 interior mutability 本身并不能自动获得线程安全。

九、一个非常实用的判断法：先问“搬走”还是“共享”

如果你在工程里遇到 Send/Sync 相关报错，我建议先不要急着背 trait 表，先问自己两句：

9.1 这个值是不是要被 move 到另一个线程？

如果是，先想 Send。

比如：

  • thread::spawn(move || ...)
  • 把任务交给工作线程
  • 把值放进跨线程 channel

9.2 这个值的引用是不是要被多个线程同时拿着？

如果是，先想 Sync。

比如：

  • 多线程共享同一份只读配置
  • 多线程共享一份带锁状态
  • 多线程共享连接池、缓存句柄等抽象对象

很多时候，这两个问题一拆开，编译错误就没那么抽象了。

十、为什么说它们是在“编译期挡数据竞争”？

Rust 最核心的目标之一，就是让数据竞争尽量在编译期被禁止。

而 Send/Sync 正是这套机制里非常关键的类型约束。

它们的价值不是让你手动调用，而是让编译器在这些场景提前拒绝危险代码：

  • 把不该跨线程 move 的值 move 过去
  • 把不该跨线程共享引用的值共享出去

这意味着很多经典并发错误，不会变成“运行久了偶发崩溃”，而会直接变成“代码过不了编译”。

这也是 Rust 并发模型最有工程价值的地方：

它不是在运行时帮你赌运气，而是在类型层面让错误更难发生。

十三、我对 Send / Sync 的几个判断

这部分是我自己的经验总结。

13.1 不要把它们背成表格，先背成两个问题

  • 值能不能搬走
  • 引用能不能共享

这比单纯记“谁是 Send 谁不是”更稳。

13.2 Arc 不是线程安全万能壳

它只解决原子引用计数，不自动解决内部数据竞争。

13.3 Mutex<T> 之所以重要，不是因为它“能改”，而是因为它把“怎么安全地改”编码进了类型里

这正是 Rust 并发抽象的价值所在。

13.4 Rc<T>、RefCell<T> 这类类型并不是“差”，只是它们的设计目标本来就是单线程

不要把 !Send、!Sync 理解成“坏类型”，而要理解成“边界不同”。

13.5 Send/Sync 的真正价值，是让线程安全约束提前变成编译错误

这比运行时再慢慢查竞争条件靠谱得多。

结语

Send 和 Sync 之所以看起来抽象，是因为它们自己不做任何事，却在并发代码里决定了很多事能不能发生。

但只要你把问题拆成这两层：

  • 这是值的跨线程转移问题
  • 还是引用的跨线程共享问题

很多原本绕的地方都会变得很顺。

所以这篇文章如果只留一句建议，我会写这句：

不要把 Send/Sync 理解成“线程安全标签”，要把它们理解成“所有权过线程边界”和“引用过线程边界”这两道不同的安全门。

这句想通了，Rc、Arc、Mutex、RefCell 这些常见类型的位置基本也就都清楚了。
