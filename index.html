深度总结Uniswap V2

发表于 2024-09-18 更新于 2024-09-19

先来看一下Uniswap v2架构图，下面是网上找的一个Uniswap v2架构图。

从图中，可以看到流动性提供者、交易者、智能合约可以与Uniswap v2 pool合约交互。下面就从这几个交互者来介绍uniswap v2。

uniswap v2介绍

uniswap是利益链上的去中心化的交易所(DEX)，V2是它的第二个版本。它的原理就是恒定乘积，每一个交易对，都是一个池子（一个Pair合约），合约中拥有两种代币，tokenA和tokenB。用户可以使用tokenA去交易tokenB，也可以使用tokenB去交易tokenA。

x表示tokenA在池子中的数量，y表示tokenB在池子中的数量。

恒定乘积就是x*y=k，这个k是恒定的，池子中的tokenA和tokenB数量都是相对变化的。

合约代码主要有两个仓库，分为：

  • v2-core合约
  • v2-periphery合约

periphery合约封装了一些方法，来调用底层的core合约。

流动性提供者

流动性提供者就是做市商，通过提供流动性给池子来获取手续费，赚取收益。它与合约交互主要有2个方法：

  • 添加流动性
  • 移除流动性

添加流动性

添加流动性，

总结：添加流动性

  • 1、计算实际需要转出的tokenA和tokenB数目
  • 2、把tokenA和tokenB转到池子pair合约
  • 3、调用pair合约的mint函数
    • 1、获取池子中之前的储量
    • 2、获取池子想在的token数量
    • 3、用现在的token数量减去之前的储量得到新添加的token数量
    • 4、通过LP token总供应量是否为0，判断是第一次添加还是非第一次添加
    • 5、第一次添加要锁住一部分流动性(转给0地址)
    • 6、计算流动性LP token数量
    • 7、mint LP token代币

流动性：

  • 第一次添加就是 tokenA数量乘以tokenB数量开平方在减去1000，因为mint了1000个LP token给0地址
  • 非第一次添加，流动性数目就是新添加tokenA数量乘以LP token总供应量除以原来tokenA数量, 新添加tokenB数量乘以LP token总供应量除以原来tokenB数量,取这里的最小值。

移除流动性

removeLiquidity函数主要把LP token转给pair合约，然后调用pair合约的burn函数。

总结：

  • 1、把LP token转给pair合约
  • 2、调用pair合约的burn方法
  • 3、burn方法：
    • 1、先计算合约中两个代币的总余额，通过balanceOf方法
    • 2、获取刚刚装给pair的LP token数目
    • 3、获取LP token的总数目
    • 4、计算拿回的tokenA、tokenB数目
    • 5、LP-toekn的数目 * 池子中代币的总余额 /LP token的总供应量
    • 6、burn LP token
    • 7、把相应数目的tokenA、tokenB转给用户
    • 8、更新池子的储备

交易者

交易者就是普通的交易用户，用一种token去池子中换取另一种token，主要就是合约的swap功能。

交易的底层调用的合约就是pair合约的swap方法，在调用到pair合约的swap方法前，它是调用的router合约，通过amountIn来计算amountOut,在把amountIn的token转给pair合约，在调用swap合约。

智能合约

智能合约部分应该是高级玩家的区域了，首先你至少会编写合约吧，能看懂合约吧。这部分主要是闪电贷和预言机的功能了。

Go语言channel总结

发表于 2024-09-14 更新于 2024-09-15

设计原理

Go语言的channel主要用于两个Goroutine之间通信的，用来收发数据。Go有一个设计里念——不要通过共享内存的方式进行通信，而是通过通信的方式来共享内存。

上图中，左边的gorutine向channel中写入，右边的gorutine从channel中读取。两个gorutine能够独立运行并不存在独立关联，但是能够通过channel间接完成通信。

数据结构

go语言runtime中，channel的数据结构是：

字段描述：

  • qcount:channel中元素个数
  • dataqsize:channel中循环队列的长度
  • buf:channel中缓冲区数据指针
  • sendx:channel中发送操作处理到的位置
  • recvx:channel中接收操作处理到的位置
  • elemsize:channel能够收发的元素大小
  • elemtype:channel能够收发的元素类型
  • sendq:当前channel由于缓冲区空间不足，而阻塞的gorutine队列（向channel发送数据的gorutine）
  • recvq:当前channel由于缓冲区空间不足，而阻塞的gorutine队列（从channel读取数据的gorutine）
  • closed:channel关闭与否的标志，1为关闭，0为未关闭

主要操作

创建

创建chan实际调用的就是runtime中的makechan函数，它实际就是新建上面hchan结构体指针。

发送数据

在发送数据操作前，会先给hchan加锁，防止多个线程来并发修改数据。如果channel已经关闭，那么向该channel发送数据会报“send on closed channel”的panic错误。

向channel发送数据可以分为以下三种情况：

  • 直接发送：存在阻塞的接收者时，直接将数据发送给阻塞的接收者
  • 写入缓冲区：缓冲区存在多余的空间，将发送的数据写入缓冲区
  • 发送数据channel进入channel本地阻塞队列：不存在缓冲区或者缓冲区满了的时候，当前发送数据的gorutine会陷入阻塞，等待其它gorutine从channel接收数据

直接发送

从channel的recvq队列中取出最先陷入等待的gorutine，并直接向它发送数据，调用下面的send函数：

send函数的执行可以分为两个部分：

  • 调用sendDirect，将发送的数据直接拷贝到接收者读取数据存储的变量的内存地址上
  • 调用runtime的goready函数，将等待接受的gorutine标记为可运行状态，在下一次调度的时候唤醒它

写入缓冲区

这里写入缓冲区的主要流程：

  • 1、先通过chanbuf函数计算出下一个可以存储的位置
  • 2、在通过typedmemmove函数将发送的数据拷贝到缓冲区中
  • 3、增加sendx和qcount计数器

因为缓冲区是一个循环队列，所以当sendx=dataqsiz时，会把sendx重置为0

阻塞发送

阻塞发送的处理流程：

  • 1、调用getg调用发送数据的gorutine
  • 2、调用acquireSudog，获取sudog结构，下面会设置这一次阻塞的相关信息，例如发送的channel、是否是select等
  • 3、将刚刚创建的sudog设置到当前gorutine的waiting上，表示该 Goroutine 正在等待该 sudog 准备就绪，在把sudog加入channel的发送等待队列中；
  • 4、调用gopark函数，将当前goroutine陷入等待唤醒状态；

这时程序是被阻塞住的，等待直到被唤醒。。。

  • 5、经过长时间等待，该gorutine被唤醒了，执行一些收尾工作，将一些属性置零并释放sudog。

读取数据

从channel接收数据，最终会映射为runtime中的chanrecv 函数：

当从一个空的channel接收数据，会直接调用gopark 让出处理器使用权；

当channel已经关闭，且缓冲区没有数据，会从清除ep指针中的数据，并立即返回；

除了上面两种情况，其它情况下，从channel接收数据可以分为三种情况：

  • 直接接收：存在等待的发送者，通过 runtime.recv 从阻塞的发送者或者缓冲区中获取数据
  • 缓冲区接收：当缓冲区存在数据时，从 Channel 的缓冲区中接收数据
  • 阻塞接收：当缓冲区中不存在数据时，等待其他 Goroutine 向 Channel 发送数据；

直接接收

当channel的sendq队列中包含处于等待状态的gorutine时，该函数会去除队列头等待的gorutine，调用recv函数来接收数据。

recv函数：

  • channel不存在缓冲区
    • 调用runtime.recvDirect将Channel发送队列中Goroutine存储的数据拷贝到目标内存地址中
  • channel存在缓冲区
    • 将队列中的数据拷贝到接收方的内存地址
    • 将发送队列头的数据拷贝到缓冲区中，释放一个阻塞的发送方

无论发生哪种情况，运行时都会调用 runtime.goready将当前处理器的runnext设置成发送数据的 Goroutine，在调度器下一次调度时将阻塞的发送方唤醒。

缓冲区接收

当 Channel 的缓冲区中已经包含数据时，从 Channel 中接收数据会直接从缓冲区中recvx的索引位置中取出数据进行处理：

当channel的缓冲区已经包含数据时，从channel中接收数据会直接从缓冲区中 recvx 的索引位置中取出数据进行处理，如果接收数据的内存地址不为空，那么会使用 runtime.typedmemmove 将缓冲区中的数据拷贝到内存中、清除队列中的数据并完成收尾工作。

收尾工作包括递增 recvx，一旦发现索引超过了 Channel 的容量时，会将它归零重置循环队列的索引；除此之外，该函数还会减少 qcount 计数器并释放持有 Channel 的锁。

阻塞接收

当channel的发送队列中不存在等待的gorutine，并且缓冲区中也不存在任何数据时，从管道中接收数据的操作会变成阻塞的。结合select语句使用就是非阻塞的。

阻塞接收使用 runtime.sudog 将当前 Goroutine 包装成一个处于等待状态的 Goroutine 并将其加入到接收队列中。

完成入队之后，上述代码还会调用 runtime.goparkunlock 立刻触发 Goroutine 的调度，让出处理器的使用权并等待调度器的调度。

关闭

channel的关闭，用关键字close函数。

它会执行runtime中的closechan 函数：

关闭流程：

  • 1、当channel是空指针或者已经被关闭了，它会panic一条信息出来
  • 2、设置channel closed字段值为1
  • 3、定义一个gList列表（一个gorutine列表）
  • 4、将recvq队列中的数据都加入到gList
  • 5、将sendq队列中的数据都加入到gList
  • 6、清空gList列表（触发调度所有的gorutine）

slice是Go语言中的可变数组，有名切片。今天来分析其底层原理。

底层数据原理

切片的底层数据有三个属性构成，指针数组、长度和容量。

  • array：指向底层数组的指针
  • len：切片的长度
  • cap：切片的容量

首先会通过切片元素大小乘以切片容量来获取内存空间大小，然后会调用mallocgc来分配内存。

内存空间=切片元素大小 x 切片容量

这里会做几个校验：

  • 计算的内存空间是否溢出
  • 内存空间是否超出了系统可分配的最大内存空间
  • 长度小于0
  • 长度大于容量

切片的扩容

当向slice追加元素，容量不足时，会触发slice的扩容操作，扩容调用的是底层的growslice函数。

growslice函数还是先获取新的扩容的容量，然后会去调用mallocgc内存分配函数，去获取一个新的内存区域，然后会调用底层的memmove函数，从oldPtr拷贝lenmem字节的数据到p指针域。

nextslicecap函数计算扩容新创建的slice的容量。它的策略是：

  • 如果新的长度大于原来容量的2倍
    • 扩容到新的长度
  • 新的长度小于等于原来容量的2倍
    • 原来容量小于256
      • 就扩容原来容量的2倍
    • 原来容量大于等于256
      • 就扩容1.25倍+192的大小，直到容量大于等于新的长度为止
